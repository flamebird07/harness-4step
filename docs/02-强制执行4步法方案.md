# 强制执行4步法方案 v3.0.0

## 核心问题

Hermes的设计是**LLM可以随时调用任何可用工具**。没有内置机制来强制执行工作流程。

## v3.0.0 实现方案

### 最终方案：Plugin + run_cli.py

**原理：** 
1. `run_cli.py` 统一执行器真实调用 CLI
2. `four-step-enforcer` 插件拦截工具调用
3. 两者配合确保每步真实执行

### 架构

```
SKILL.md 定义调用规范
        ↓
scripts/run_cli.py 统一执行真实 CLI
        ↓
CLI stdout/stderr + evidence.json
        ↓
four-step-enforcer 验证证据
        ↓
满足前置步骤后才允许进入下一步/修改文件
```

### 实现细节

#### 1. run_cli.py（统一执行器）

```python
STEP_CLI = {
    "step1": {"agent": "codex", "executable": "codex.cmd", ...},
    "step2": {"agent": "codex", "executable": "codex.cmd", ...},
    "step3": {"agent": "mimo", "executable": "codex.cmd", "--model": "mimo", ...},
    "step4": {"agent": "kimi", "executable": "kimi.cmd", ...},
}
```

- 真实调用 CLI 进程
- 捕获 stdout/stderr
- 生成 evidence.json
- 支持 `--verify-only` 模式

#### 2. enforcer 插件（拦截器）

```
pre_tool_call hook:
    delegate_task → Allow (但不标记step_done)
    terminal+run_cli.py → Allow
    write_file/patch → 检查step1_done，未完成则阻塞

post_tool_call hook:
    delegate_task → 不标记step_done
    terminal+run_cli.py → 仅在成功时标记step_done
```

### 关键变更

| v2.x | v3.0.0 |
|------|--------|
| delegate_task 标记step1_done | delegate_task 不再标记 |
| 角色标签冒充真实CLI | 必须真实调用CLI |
| 无证据验证 | evidence.json 证据系统 |

## 之前的方案（已废弃）

### 方案1：修改Hermes源代码
- ❌ 每次升级可能被覆盖
- ❌ 需要维护自定义版本

### 方案2：approvals.mode: manual
- ❌ 太烦，每次操作都要确认

### 方案3：独立Profile
- ❌ 需要用户手动切换profile

### 方案4：强化System Prompt
- ❌ 靠自觉，不可靠

## 推荐方案

**方案：Plugin + run_cli.py（已实现）**
- 不修改Hermes核心代码
- 可以独立升级
- 真实CLI执行 + 证据验证
- 技术层面强制执行

## 文件位置
