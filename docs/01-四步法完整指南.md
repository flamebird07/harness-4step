# 4步法完整指南 v3.0.0

## 一、核心原则

**裁判不能当运动员，审查的人不能同时写代码，分权制衡才能发现问题。**

## 二、v3.0.0 重大升级：真实CLI执行

### 问题背景

v2.x 版本中，`delegate_task` 的 goal 字段标注了 `【角色：Codex CLI】` 等角色名，但实际执行的是 Hermes 内置的 `mimo-v2.5` 模型，**并非真正的 Codex CLI 或 Kimi CLI**。这是角色欺骗。

### v3.0.0 解决方案

| 变更 | 说明 |
|------|------|
| 新建 `run_cli.py` | 统一CLI执行器，真实调用 codex/kimi/mimo CLI |
| 更新 `SKILL.md` | 加入 CLI Execution Contract，禁止角色标签冒充 |
| 更新 enforcer 插件 | delegate_task 不再标记 step1_done，需真实CLI调用 |

### CLI Execution Contract

**每步必须通过 `run_cli.py` 调用真实 CLI：**

```bash
python scripts/run_cli.py \
  --step step1 \
  --task-id <unique-task-id> \
  --workspace <project-root> \
  --prompt "<step-specific-prompt>"
```

**禁止行为：**
- ❌ 用 `delegate_task` 返回文本冒充 CLI 输出
- ❌ 手工创建 `evidence.json`
- ❌ CLI 失败后继续推进步骤

**完成条件：**
1. `exit_code: 0`
2. `stdout` 非空
3. `evidence.json` 存在且 `success: true`

## 四、四步法流程

### 第1步：Codex CLI 审查（只找问题，不写方案）
- **角色**：Codex CLI（真实调用 `codex exec`）
- **输入**：用户描述的问题 + 相关代码文件路径
- **输出**：问题清单 [{文件, 行号, 现象, 影响}]
- **禁止**：不能给修复建议、不能给 before/after 代码

### 第2步：Codex CLI 出方案（只写计划，不改代码）
- **角色**：Codex CLI（真实调用 `codex exec`）
- **输入**：第1步的问题清单
- **输出**：修复方案 [{文件, 行号, before代码, after代码}]
- **禁止**：不能执行修改、不能跳过问题清单直接写方案

### 第3步：MiMo Code 执行（只改代码，不分析）
- **角色**：MiMo Code（真实调用 `codex exec --model mimo`）
- **输入**：第2步的修复方案
- **输出**：修改后的代码文件
- **禁止**：不能自己决定改什么、不能分析问题

### 第4步：Kimi CLI K3 复审（只验证，不改代码）
- **角色**：Kimi CLI K3（真实调用 `kimi -p`）
- **输入**：第3步修改后的代码 + 原始问题清单
- **输出**：修复评级（通过/需调整）+ 遗漏问题
- **禁止**：不能自己修复、不能跳过验证

## 五、Loops循环机制

### 循环规则
- 第4步复审不通过 → 回到第2步 → Codex CLI修复 → 再次复审
- 最大循环次数：10次
- 10次循环后仍FAIL，报告给用户决定是否继续

### 循环流程
```
Loop 0: Step1 → Step2 → Step3 → Step4 (FAIL)
         ↓
Loop 1: Step2(新问题) → Step3 → Step4 (FAIL)
         ↓
Loop 2: Step2(新问题) → Step3 → Step4 (PASS或FAIL)
```

## 六、韧性执行机制

### 核心铁律
**中途产生的问题不能打断总进程，必须完成最初的目标。**

### 执行原则
1. **目标锁定**：任务开始时明确记录「最初目标」
2. **失败处理**：遇到问题不中断，换方案继续
3. **至少尝试3种方法**后再考虑报告用户

### 绝对禁止
- ❌ 遇到一个错误就停止整个任务
- ❌ 说"做不到"而不尝试替代方案
- ❌ 把问题抛回给用户而不给出已尝试的方案
